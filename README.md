# Инструкция пользователя

## Главная страница

### Выбор user`а
На главной странице есть возможность создать датасеты в CSV формате выбрав одного из user`ов. 
Если вы загрузили свои БД с большим количеством записей и с теми же полями, то для вас есть возможность выбрать user при помощи ввода и/или поиска среди выданных значений. 

### Получение Сsv датасетов
После выбора пользователя, но которого вы хотите получить датасет, нажимаете "Показать данные". После этого вам покажет собранные в таблицы данные. Под данными таблицами можно нажать "Скачать датасет", после чего датасет, под которым вы нажали кнопку, пудет скачан на ваш ПК в CSV формате

## API
### Для получения данных в виде JSON вы можете выполнить две команды
Для получения данных о том, сколько и под каким постом пользователь писал комментарии:
```
GET http://127.0.0.1:8000/api/comments?login=<userloggin>
```
Для получения общей информации о действиях пользователя:
```
GET http://127.0.0.1:8000/api/general?login=<userloggin>
```

Вместо `<userloggin>` пишите логин пользователя, которого вы ищете. Например user1: 

```
GET http://127.0.0.1:8000/api/general?login=user1
```

### Так же вы можете просмотреть выданные данные в браузере. Просто не пишите `GET`
Для получения данных о том, сколько и под каким постом пользователь писал комментарии:
```
http://127.0.0.1:8000/api/comments?login=<userloggin>
```
Для получения общей информации о действиях пользователя:
```
http://127.0.0.1:8000/api/general?login=<userloggin>
```

## 2. В одной бд (db1_scheme.png) создать три таблицы - post, author и blog. 
В первой должна содержаться информация о посте (id, header, text, author_id, blog_id). 

Во второй информация об авторе (id, login, email).

В третьей информация о блоге (id, owner_id, name, description).

таблица post связана через внешние ключи с таблицами user (author_id) и blog (blog_id),

таблица blog связана через внешний ключ с таблицей user (owner_id)

## 3. Во второй бд (db2_scheme.png) создать три таблицы - logs, space_type и event_type. 
В первой таблице будут содержаться действия пользователя (id, datetime,user_id, space_type_id (внешний ключ для связи с таблицей space_type), event_type_id (внешний ключ для связи с таблицей event_type), space_id (id пространства, в котором было совершено действие)

user_id может содержать идентификаторы пользователи из разных БД, предполагается что в том числе и из нашей первой созданной

## space_type содержит типы пространств где пользователь может совершать действия, пространства могут быть такими:
- global

- blog

- post

## event_type содержит типы действий: 

- login

- comment

- create_post

- delete_post

- logout

Предполагается, что действия типа login, logout выполняются в пространстве global, действия типа create_post, delete_post выполняются в пространстве blog, действие comment выполняется в пространстве post

# Проделанные правки в условии
Во второй базе данных в таблице logs было добавлено поле Space_id, в котором хранится id пространства, в котором было проведено действие. 
Если Space_type - global, то атрибут данного значения Null; blog - id блога, в котором проводилось действие (создание/удаление поста); post - id поста по аналогии с blog

Была идея создать таблицу comment (id, author_id, post_id, text) в первой БД (и даже реализована), но я её отсёк по причине одного из основных условий - конект 2 БД. В случае её создания, первый датасет можно было сделать без использования 2 БД, путем поиска данных только внутри БД - user_id ищется по логину внутри неё же, id поста, под которым бsл создан комментарий и поиск всех необходимых данных, ну и подсчёт самих элементов comment. 

Поэтому решил создать space_id. В данном решении в API запрос подается login пользователя, по нему находится user_id, затем в logs проходит фильтрация по нему и event_type_id для поиска в логах действия comment. После чего через space_id достаются посты, в которых было совершено действие comment, и достаются header и author_id. По author_id ищется логин автора. После уже идет подсчёт для каждого поста совершенных действий comment user`ом. 

# О выбранном решении
Для выполнения данной задачи был выбран фреймворк Django(DRF). В нем встроена ORM система, которая посзволяет удобно работать с базами данных. Django Rest Framework же позволяет удобно создавать API

Внутри проекта было реализованно 3 приложения: blogs, logs и UserActions. В первых двух созданы модели для соответствующих БД и роутеры, для привязки моделей к ним. В UserActions происходит основная 
работа с ними. В нем созданы сериализаторы для представления данных в JSON. Так же в views.py прописанны функции для выдачи данныъ по API запросу, функции для вывода этих данных на главную страницу и функция для 
преобразования выданного по API запросу JSON файла в датасет в CSV формате.
